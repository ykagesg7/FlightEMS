## 1. プロジェクトアイデンティティ

### 名称
**Flight Academy**（リポジトリ名: FlightAcademyTsx）

### ビジョン
> 日本語圏で最も実用的な航空学習プラットフォームを構築し、航空の世界への入り口を提供する。

### コアコンセプト: "Learn, Plan, Fly"

| ステップ | 意味 | 対応機能 |
|:---------|:-----|:---------|
| **Learn** | 航空知識を体系的に学ぶ | PPL/CPL 学習コンテンツ、クイズ、SRS |
| **Plan** | 実践的なフライトプランを作る | Flight Planning Tool、航空気象 |
| **Fly** | 知識を持つ仲間として飛び立つ | ゲーミフィケーション、コミュニティ |

### プラットフォームの位置づけ
- **独立運営**: 外部の承認や許可に依存せず、自走可能な開発を行う
- **パートナーシップにオープン**: エアロバティックチーム、フライトスクール等との連携は将来のオプション
- **コア機能はパートナーに依存しない**: パートナーシップは拡張レイヤーであり、必須要件ではない

### 戦略転換の経緯（2026年2月）

本プロジェクトはもともとパイロット学習アプリとして開発が始まり、その後エアロバティックチーム「Whisky Papa」の非公式ファンサイトとして方向転換した。しかし以下の問題が顕在化した：

- パイロット紹介、Shop、体験搭乗（Experience）など主要機能が**外部の承認待ちでブロック**
- 公式/非公式の立場が不確定で、コンテンツの半分近くが非表示またはコメントアウト状態
- 開発の推進力が外部要因に依存し、**足踏み状態が長期化**

この状況を打破するため、**Flight Academy として独立したプラットフォームに回帰**し、Whisky Papa との関係は将来的なパートナーシップの一オプションとして位置づける「ハイブリッド戦略」を採用する。

## 2. ターゲットユーザー

優先順位を明確にし、リソースの集中先を定める。

### Tier 1: PPL/CPL 学科試験受験者（最優先）
- 自家用操縦士（PPL）学科試験の受験予定者
- 事業用操縦士（CPL）学科試験の受験予定者
- フライトスクール在籍者
- **ニーズ**: 体系的な学習コンテンツ、試験問題演習、進捗管理
- **規模**: 日本国内で年間数千人規模

### Tier 2: フライトシミュレーター愛好家
- Microsoft Flight Simulator、X-Plane 等のユーザー
- リアルな航法・気象知識を求めるシマー
- **ニーズ**: フライトプランニング、航空気象情報、航法知識
- **規模**: 日本国内で数万人規模

### Tier 3: 航空ファン・エアショー来場者
- 航空祭来場者
- 航空関連コンテンツの愛好者
- **ニーズ**: 航空知識、コミュニティ、ギャラリー
- **規模**: 潜在的に大きいが、コンバージョン率は低い

## 3. 戦略4本柱

```
Flight Academy
├── [1] Content   ─ プラットフォームの核心価値
├── [2] Tools     ─ 差別化要素
├── [3] Community ─ エンゲージメント
└── [4] Partnership ─ 将来の拡張オプション
```

### 柱1: Content（学習コンテンツ）— 核心価値

プラットフォームの存在意義そのもの。ユーザーが訪れる理由の第一。

| 項目 | 現状 | 目標 |
|:-----|:-----|:-----|
| PPL 記事 | 15/150（10%） | 75/150（50%）— 2026年末 |
| CPL 記事 | 0/19（Phase 1） | Phase 1 着手 — 2026年末 |
| MDX 記事合計 | 48 ファイル | 100+ ファイル — 2026年末 |

**方針**:
- PPL Master Syllabus（150記事）に従い、Phase 1（高頻出・重要トピック）から優先的に記事化
- AI アシスト制作パイプラインを確立し、月5記事ペースを目指す
- 品質基準は `docs/07_PPL_Master_Syllabus.md` および `docs/06_記事作成ロードマップ.md` に従う

### 柱2: Tools（航空ツール）— 差別化要素

他の学習サイトにはない、実用的なツール群。

| ツール | 状態 | 説明 |
|:-------|:-----|:-----|
| Flight Planning Tool | 実装済み | 経路計画、測地線計算、空域判定、燃料計算、A4印刷 |
| Quiz / SRS | 実装済み | CPL 試験問題、間隔反復学習 |
| 航空気象（METAR/TAF） | 実装済み | NOAA API 統合、リアルタイム気象データ |
| 一般気象 | 実装済み | WeatherAPI.com 統合 |

**方針**:
- 新規ツール開発よりも既存ツールの安定化・品質向上を優先
- テストカバレッジの拡充（特に `airspace.ts`, `planDocument.ts`, `offset.ts`）
- ユーザーフィードバックに基づく改善サイクルの確立

### 柱3: Community（コミュニティ）— エンゲージメント

ユーザーの継続利用と関与を促進する仕組み。

| 機能 | 状態 | 説明 |
|:-----|:-----|:-----|
| ゲーミフィケーション | 実装済み | ランク、XP、ミッション、ストリーク、達成通知 |
| Gallery | 実装済み | 航空写真投稿・イベント管理 |
| コメント | 実装済み | 記事へのコメント機能 |
| ランキング | 未実装 | XP・正答率ベースのランキング |

**方針**:
- ゲーミフィケーションは現行のまま維持（ランク名・XP体系は変更不要）
- Gallery はチーム専用から航空写真コミュニティへ方向転換
- ランキング機能は Phase D（2026年後半）で実装

### 柱4: Partnership（パートナーシップ）— 将来オプション

コア機能が安定し、ユーザーベースが確立した後に追求する拡張レイヤー。

| パートナー候補 | 連携内容 | 前提条件 |
|:---------------|:---------|:---------|
| Whisky Papa | コラボセクション（チーム紹介、Shop、体験搭乗） | 公認取得 |
| フライトスクール | 教材連携、受講者誘導 | 実績・ユーザーベース |
| 航空書籍出版社 | 書籍紹介、Amazonアソシエイト | コンテンツ充実後 |
| フライトシム関連 | ツール連携、コミュニティ | ユーザーベース |

**重要な原則**: パートナーシップはプロジェクトの推進力を加速するものであり、ブロッカーにしてはならない。公認や承認を「待つ」のではなく、プラットフォームの価値を高めてから交渉に臨む。

## 4. ゲーミフィケーション設計（進化版）

### ランクシステム（変更なし）

現行のランクシステムはそのまま維持する。コードやデータベースの変更は不要。

**基本ランク（XPベース）:**
1. **Fan** (0 XP) — 新規ユーザー

**PPL 中間ランク（記事完了ベース）:**
2. PPL 空力基礎初級 〜 PPL 航空工学マスター — 段階的に取得

**上位ランク:**
10. **PPL** (500 XP + 全 PPL 記事読了) — 自家用操縦士
11. **Wingman** (1,200 XP) — 一緒に飛べる知識を持つ仲間
12. **CPL** (1,000 XP + 全 CPL 記事読了) — 事業用操縦士
13. **Ace** (1,500 XP) → **Master** (2,000 XP) → **Legend** (2,500 XP)

### Wingman の再解釈

| 旧解釈（Whisky Papa） | 新解釈（Flight Academy） |
|:----------------------|:------------------------|
| エアロバティックチームの僚機 | 航空知識を持ち、仲間と一緒に飛べるレベルに達した学習者 |
| チームへの帰属 | 航空コミュニティへの貢献 |
| 体験搭乗の資格 | 高度な学習達成の証 |

### XP・ミッション・ストリーク（変更なし）
- 記事読了 XP、シリーズ完了ボーナス、ストリーク倍率 — 全て現行どおり
- ミッションタイプ（one_time, daily, weekly）— 変更なし
- 技術的な変更は不要

## 5. パートナーシップモデル

### Whisky Papa パートナーシップ（将来オプション）

**現在の状態**: 非公式ファンサイトとして運営していたが、コア機能がブロックされたため独立路線に転換

**パートナーシップ成立時の復活項目**:
- About ページ: パイロット紹介セクションの復活
- Shop: Hangar Store の復活（ランク連動型物販）
- Experience: 体験搭乗予約機能の実装
- Schedule: マーケティングセクションの復活

**パートナーシップ成立までの扱い**:
- コメントアウト状態のコードはそのまま保持（削除しない）
- `/shop` → ホームへのリダイレクトは維持
- `/experience` → `/mission` へのリダイレクトは維持
- ドキュメント上は「パートナーシップ機能」として分類

**パートナーシップ交渉の前提条件**:
1. Flight Academy として安定稼働していること
2. PPL 記事が50%以上完了していること
3. 月間アクティブユーザーが存在すること
4. 上記の実績を持って交渉に臨む

### その他のパートナーシップ

将来的に以下の連携を検討する（Phase E 以降）:
- フライトスクール: 教材としての採用、学生向け割引プログラム
- 航空書籍出版社: 書評記事、Amazonアソシエイトリンク
- フライトシム関連: MSFS/X-Plane 向けフライトプラン連携

- CI/CD（GitHub Actions）
- GeoJSON データ

### 移行のタイミング

ブランド移行は **Phase C（2026年6月〜）** で実施する。Phase A/B ではコンテンツとテストに集中し、見た目の変更は後回しにする。

## 7. 現状評価（2026年2月スナップショット）

### 強み
- **航空専門機能の完成度**: Flight Planning（測地線計算、空域判定、燃料計算、A4印刷）は他に類を見ない
- **ゲーミフィケーション設計**: PPL 中間ランク + XP ベースランクの統合システムは精巧
- **技術基盤**: TypeScript strict モード、ESLint any 禁止、React 18、Vite によるモダンな構成
- **ドキュメント整備**: 15+ の詳細ドキュメント、AI 協働開発に最適化

### 弱み
- **テストカバレッジ: 3.9%**（6ファイル/39テスト vs 155ソースファイル）— 目標50%に対して大幅未達
- **コンテンツ: 10%**（PPL 15/150記事）— プラットフォームの核心価値が未充足
- **運用監視: なし** — エラー監視、アクセス解析、パフォーマンス監視が未導入
- **ユーザー: 0** — 実際のユーザーが不在のため、フィードバックループが存在しない
- **ソロ開発**: Bus Factor 1 のリスク

### ブロッカー解消
- **旧**: パイロット紹介、Shop、Experience が外部承認待ちでブロック
- **新**: Flight Academy として独立運営。上記はパートナーシップ成立時のオプションに格下げ

## 8. 開発における意思決定基準

### 機能追加の判断基準

新機能を追加する前に、以下を確認する:

1. **コンテンツとテストは進んでいるか？** — PPL 記事とテストカバレッジが最優先。新機能はこれらが計画通り進んでいる場合のみ検討
2. **コア機能の安定化に寄与するか？** — 既存機能の品質向上（テスト、エラーハンドリング、パフォーマンス）が新機能より優先
3. **パートナーシップに依存していないか？** — 外部承認が必要な機能は Phase E 以降に分類

### 優先順位の原則

```
コンテンツ制作 > テスト拡充 > 既存機能の安定化 > 運用基盤 > 新機能 > ブランド変更
```

この順序は Phase A-B の間は厳格に守る。Phase C 以降はブランド移行を含む。

## 9. ディレクトリ構成（変更なし）

ハイブリッド戦略への転換にあたり、ディレクトリ構成の変更は不要。

```
src/
├── components/          # 共通コンポーネント
│   ├── ui/             # デザインシステム（Button, Card, Typography）
│   ├── marketing/      # マーケティング共通（RankBadge, MissionCard）
│   ├── mdx/            # MDX関連（MDXLoader, MDXContent）
│   ├── ppl/            # PPLランク関連（PPLRankBadge, PPLRankList）
│   └── common/         # 汎用ユーティリティコンポーネント
├── pages/              # ページコンポーネント（ハイブリッド方式）
├── hooks/              # カスタムフック（14ファイル）
├── stores/             # Zustandストア（authStore）
├── utils/              # ユーティリティ関数（21ファイル）
├── types/              # TypeScript型定義（15ファイル）
├── layouts/            # レイアウト（MarketingLayout）
├── content/            # MDX記事コンテンツ（48ファイル）
├── contexts/           # React Context（Progress, WeatherCache）
├── services/           # 外部サービス統合
└── styles/             # テーマ定義
```
