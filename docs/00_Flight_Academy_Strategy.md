## 1. プロジェクトアイデンティティ

### 名称

**Flight Academy**（リポジトリ名: FlightAcademyTsx）

### ビジョン

> 日本語圏で最も実用的な航空学習プラットフォームを構築し、航空の世界への入り口を提供する。

### コアコンセプト: "Learn, Plan, Fly"


| ステップ      | 意味             | 対応機能                      |
| --------- | -------------- | ------------------------- |
| **Learn** | 航空知識を体系的に学ぶ    | PPL/CPL 学習コンテンツ、クイズ、SRS   |
| **Plan**  | 実践的なフライトプランを作る | Flight Planning Tool、航空気象 |
| **Fly**   | 知識を持つ仲間として飛び立つ | ゲーミフィケーション、コミュニティ         |


### プラットフォームの位置づけ

- **完全独立運営**: 外部の承認・パートナー契約・特定団体の公認に依存せず、自走可能な開発とプロダクト判断を行う
- **外部承認が前提の機能は採用しない**: 戦略ロードマップに「承認待ち」を置かない

### 戦略転換の経緯（要約）

本プロジェクトはパイロット学習アプリとして始まり、一時期エアロバティックチーム向けファンサイト色が強まったが、外部要因で主要機能がブロックされ開発が停滞した。**Flight Academy として独立した学習プラットフォームに回帰**し、以降は上記のとおり独立運営のみを原則とする。

## 2. ターゲットユーザー

優先順位を明確にし、リソースの集中先を定める。

### Tier 1a: 事業用操縦士（CPL）学科試験受験者（最優先・主軸）

- CPL 学科試験の受験予定者、フライトスクール在籍者のうち学科試験が要件となる層
- **ニーズ**: 体系化された **CPL 記事**、クイズ・SRS（**試験範囲は PPL のみ / CPL 等で選択可能**）、進捗管理、自学の効率化
- **ニーズ（追加の観点）**: 組織内の試験対策が限定的な場合、自学の比重が大きくなりがちである。当プラットフォームは**いかなる組織・団体の公式サービスでもなく**、個人による学習支援（コンテンツ・ツール）である。
- **規模**: 日本国内で年間数千人規模

### Tier 1b: 自家用操縦士（PPL）学科試験受験者（最優先・継続）

- PPL 学科試験の受験予定者
- **ニーズ**: **PPL 記事**（出題範囲に沿った本編）、同じくクイズの **PPL 範囲フィルタ**、進捗管理
- **リソース配分**: 記事の**執筆・シラバス上の優先順位は CPL を先に**進め、PPL 記事は Master Syllabus に沿って継続整備する。CPL 記事から PPL 記事へのリンクで基礎復習を行う設計とし、**PPL 記事は CPL 受験者のオプション経路としても価値を持つ**

### Tier 2: フライトシミュレーター愛好家

- Microsoft Flight Simulator、X-Plane 等のユーザー
- リアルな航法・気象知識を求めるシマー
- **ニーズ（補足）**: DCS 等、**軍用機・戦術系シミュレーター**に関心がある層は、フライトプランニング・航空気象・航法ツールとの**親和性**から**二次的ターゲット**になりうる。**サイト全体のミリタリー特化は行わない**（民用学習・ツールが主軸）。
- **ニーズ**: フライトプランニング、航空気象情報、航法知識
- **規模**: 日本国内で数万人規模

### Tier 3: 航空ファン・エアショー来場者

- 航空祭来場者
- 航空関連コンテンツの愛好者
- **ニーズ**: 航空知識、コミュニティ、ギャラリー
- **規模**: 潜在的に大きいが、コンバージョン率は低い

### 価値命題：事業用学科（CPL）と自学の効率

- **対象**: 事業用操縦士**学科試験**（民航の試験制度）に向けた学習。
- **提供する価値**: 体系化された記事、クイズ、SRS、進捗管理により、教材・問題集を中心とした自学に伴う **学習順序の探索・範囲の設定・反復の設計** にかかる**手間と時間**を下げる。
- **スコープの限定**: **職務訓練・運用訓練・組織固有の教育の代替ではない**。**民航の事業用操縦士学科試験要件の充足**にフォーカスする。

## 3. 持続可能性・運営上の原則（個人開発）

### 3.1 収益とキャッシュフロー

- 運営は**個人開発**を前提とし、当初の経済目標を **開発ツール・外部 API・インフラ等の固定費・変動費を賄えるキャッシュフロー** に置く。
- **収益が固定費を上回った段階**で、API 利用枠の拡大、監視・品質、コンテンツ制作、テスト拡充など**プロダクト強化へ再投資**する。
- 具体的な課金モデル・実装は [01_Current_Status_and_Roadmap.md](01_Current_Status_and_Roadmap.md) および将来の別紙に委ね、本書では**原則のみ**を定義する。

### 3.2 運営者の専門性と公開の粒度

- **航空運航に関する実務的な専門性**は、コンテンツの信頼性・差別化の源泉として位置づける。
- **所属・肩書き・実名の公開粒度**: 公開リポジトリ内のドキュメントでは**原則として抽象表現**（例: 現職パイロット等）にとどめる。**ウェブサイトの About 等での具体表記は、所管における確認・許容範囲を踏まえて判断する**。個人名・所属の記載は**必須ではない**。

### 3.3 コンプライアンス（開発・マーケ時のチェックリスト）

本項は**法的助言ではない**。運用前に**所属その他の所管**で必ず確認すること。

- **兼業・副業**: 開始・変更の前に、**規程に基づく届出・承認**を完了させる。
- **サービスの位置づけ**: **個人・非公式**の学習支援であることを明示し、**公式見解や特定組織の代言**に読まれない表現にする。**規程確認前にマーケティング文言を確定しない**。
- **コンテンツ**: **公開情報・学科試験の範囲**に寄せ、機密・内部手続き・識別に資しうる情報を扱わない。

## 4. 戦略3本柱

```
Flight Academy
├── [1] Content   ─ プラットフォームの核心価値
├── [2] Tools     ─ 差別化要素
└── [3] Community ─ エンゲージメント
```

### PPL 記事と CPL 記事の棲み分け（Content の前提）

- **記事は PPL 用と CPL 用で分離する**（URL・メタデータ・主たる読者が異なる）。
- **原則**: PPL 受験者は **PPL 記事**を、CPL 受験者は **CPL 記事**を主に読む。
- **CPL 記事**: CPL 出題範囲にフォーカスし**深く**書く。PPL と重なる基礎は本文では**要約にとどめ**、**PPL 記事へのリンク**で補完する。
- **CPL 受験者**が根本理解で詰まったとき、**任意で PPL 記事へ遷移**して復習できる動線を想定する（リンク・Callout 等）。
- **共通の事実・定義・法規の更新**: 原則として **PPL 記事側を正本**とし、CPL 側は要約・リンクに合わせて追随する（二重の長文を避ける）。
- **クイズ**: `/test` 等で **PPL のみ / CPL 範囲**を選べる機能は**維持**する（試験制度上、記事の主経路とクイズのスコープを対応させる）。

プラットフォームの存在意義そのもの。ユーザーが訪れる理由の第一。

| **CPL 記事** | **Phase 1 本文化 19/19 完了**（2026-04-12。正本 [db/CPL_KPI_and_Database_Operations.md](db/CPL_KPI_and_Database_Operations.md)）。計測の補助: [14](Article_Coverage_Backlog.md) | **以降の最優先**は [05](05_Content_Pipeline.md) **Phase 2** および**マッピング精緻化**（[01](01_Current_Status_and_Roadmap.md) 整合） |

### 柱2: Tools（航空ツール）— 差別化要素

他の学習サイトにはない、実用的なツール群。


| ツール                  | 状態   | 説明                        |

| 機能         | 状態   | 説明                                                 |

## 9. 開発における意思決定基準

### 機能追加の判断基準

新機能を追加する前に、以下を確認する:

1. **コンテンツとテストは進んでいるか？** — **CPL 記事**（および PPL 記事のリンク整備）、**テストカバレッジ**が計画に沿っているか。新機能はこれらが計画通り進んでいる場合のみ検討
2. **コア機能の安定化に寄与するか？** — 既存機能の品質向上（テスト、エラーハンドリング、パフォーマンス）が新機能より優先
3. **外部承認・特定パートナーに依存しないか？** — **依存する機能は採用しない**（ロードマップに「承認待ち」を置かない）
4. **外部 API・従量コストは採算と整合するか？** — 利用拡大時はキャッシュ・認可範囲・上限設計を [01](01_Current_Status_and_Roadmap.md) の持続可能性タスクと照合する

### 優先順位の原則

```
コンテンツ制作（CPL 主軸・PPL 継続） > テスト拡充 > 既存機能の安定化 > 運用基盤 > 新機能 > ブランド変更
```

この順序は Phase A-B の間は厳格に守る。Phase C 以降はブランド移行を含む。
