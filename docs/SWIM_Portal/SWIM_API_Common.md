## 1. はじめに

本ドキュメントは、「SWIM サービスの API 連携利用」に関する仕様書である。SWIM サービス利用者が SWIM サービス提供システム（以後、「**本システム**」と記載する）に対して API 連携する方法を記載する。本ドキュメント及び SWIM サービスの API 連携の利用者は、以下のプロトコル等の知識があることを前提とする。

- HTTP
- AMQP
- XML
- JSON

SWIM サービスの API 連携利用については、**WebAPI** と **AMQP** の 2 つの方法がある。提供する SWIM サービスにより通信方式が異なるので、詳細は **3. 提供サービス** の **別紙 1 提供サービス一覧** 及び各付録を参照すること。

- **WebAPI 連携:** HTTP（REST）通信によるリクエストをサーバへ送信し、そのリクエストに応じたレスポンスをサーバから受信する。
- **AMQP 通信によるデータ配信:** サービス提供者側のイベント発生時に行われる。データの受信には事前に受信設定が必要となる。

## 3. 提供サービス

本システムが提供するサービスについて、WebAPI、AMQP のサービス提供方式について記載する。提供サービス一覧は **別紙 1** に記載する。

### 3.1 サービス提供方式（WebAPI）の説明

#### 3.1.1 概要

本章では本システムの WebAPI で使用する、本システムとサービス利用者間のネットワーク接続インタフェースの共通的な部分について規定する。

本システムとサービス利用者との接続は、TCP/IP で接続する。

```text
[サービス利用者] --- (インターネットまたは VPN 等 TCP/IP) ---> [本システム]
```

#### 3.1.2 アプリケーション層

アプリケーション層の仕様を **表 3.1.2-1** に示す。

本仕様書では本システムで特に規定する項目のみを記載する。以下に記載のない項目については、一般的な HTTP の仕様に準じるものとする。

**表 3.1.2-1 アプリケーション層の仕様**

| 項番 | 項目             | 適用 |
|------|------------------|------|
| 1    | プロトコル       | HTTP バージョン 1.1（RFC 9110・9111・9112 に準拠） |
| 2    | 論理回線の冗長化 | 特に規定しない。 |
| 3    | 文字コード       | HTTP バージョン 1.1 に準じる。 |
| 4    | 通信シーケンス   | HTTP バージョン 1.1 に準じる。 |
| 5    | メッセージフォーマット | HTTP バージョン 1.1 に準じる。 |
| 6    | 最大メッセージ長 | HTTP バージョン 1.1 に準じる。 |

アプリケーション層で使用する HTTP のメッセージ構成は一般的な HTTP の仕様に準ずる（HTTP リクエスト／ステータス、HTTP ヘッダ、HTTP ボディ）。

本システムは、サービス利用者に対してサーバ側の位置付けとなる。

**表 3.1.2-2 HTTP の仕様詳細（ログインサービスによる認証後の仕様）**

| 項番 | 項目           | 適用 |
|------|----------------|------|
| 1    | HTTP バージョン | 1.1（RFC 9110・9111・9112、RFC 2616 に準拠） |
| 2    | HTTP メソッド  | 各付録に記載 |
| 3    | HTTP ヘッダ    | 下記参照 |
| 4    | HTTP ボディ    | クライアントおよびサーバ側の設定を踏襲する |
| 5    | HTTP ステータス | 詳細を「3.1.2.1.4 ステータスコード」に示す |

**項番 3 HTTP ヘッダ（表 3.1.2-2 補足）**

- `Set-Cookie` / `Cookie`: **MSMSI**, **MSMAI**
- `Domain`: `mlit.go.jp`
- `HttpOnly`: 有効
- `Cache-Control`: `private, no-cache`
- `Connection`: `Keep-Alive`
- `Keep-Alive`: `timeout=15, max=100`

本システムにおいては、Cookie に認証情報を格納し、認証・認可やシングルサインオン、アクセス制御等を実現する。サービス利用時にはログイン時に取得した Cookie（MSMSI、MSMAI）をセットすること。

インターネットを介して通信を行うサービス利用者の環境は、さまざまな環境が想定され、企業内などのプロキシサーバを経由した通信の中継を行うことが多いと想定される。

そのため、インターネット上のプロキシサーバにはキャッシュ（共有キャッシュ）を許可しない。ただし、本システムを介してサービス提供を行うシステムのサービスによっては、静的なファイルをキャッシュすることで通信効率の向上に繋がるため、クライアント側でのキャッシュ（プライベートキャッシュ）を許可する。

また、クライアント側でキャッシュされたデータに対し、最新のデータであるかの再検証を要求する。ただし、特定バージョンを示す識別子（ETag）、更新時刻を示す情報（Last-Modified）、有効期限（max-age）などキャッシュ有効を示す情報は付与しない。

通信の効率化と性能向上を目的として HTTP の Keep-Alive を有効にする。

**表 3.1.2-3 詳細エラーステータスコード一覧**

| エラーコード | エラーメッセージ           | 意味 |
|--------------|----------------------------|------|
| 400          | Bad Request                | リクエストが不正 |
| 401          | Unauthorized               | 認証が必要 |
| 403          | Forbidden                  | アクセス禁止、Cookie が無効（未設定 or 有効期限切れ等） |
| 408          | Request Timeout            | リクエストタイムアウト |
| 410          | Gone                       | 消滅 |
| 500          | Internal Server Error      | サーバ内部エラー |
| 503          | Service Unavailable        | サービス利用不可 |

リバースプロキシサーバでは HTTP ヘッダサイズの上限値は **8KB** となる。このサイズを超過した通信はリバースプロキシサーバで `403 Forbidden` を返却する。

**表 3.1.2-4 セッションタイムアウト**

| 項番 | タイムアウト             | タイムアウト発生時の挙動 |
|------|--------------------------|---------------------------|
| 1    | 無通信タイムアウト       | Cookie が無効となり、サービス利用者に `403 Forbidden` が返却される。ブラウザは、SWIM ポータルのログイン画面にリダイレクトされる。 |
| 2    | 強制セッションタイムアウト | 同上 |

- **無通信タイムアウト:** 一定時間サービス利用者から本システムへの通信が発生しなかった場合に、認証情報を無効にする。
- **強制セッションタイムアウト:** サービス利用者から本システムへの通信が継続的に発生していた場合でも一定時間経過後に強制的に認証情報を無効とし、再度ログイン認証を行わせる。長時間通信を継続している場合に権限変更等を認証情報に反映するために行う。

#### 3.1.3 通信シーケンス

WebAPI で使用する HTTP 通信は、一般的な通信シーケンスであるため本仕様書では特に規定しない。

### 3.2 サービス提供方式（AMQP）の説明

#### 3.2.1 概要

本章では本システムの AMQP で使用されるシステム間ネットワーク接続インタフェースおよび、システム間通信の伝送制御プログラム作成に必要なサービスの共通的な部分について規定する。

本システムと SWIM サービス利用者システム間は、TCP/IP で接続される。

#### 3.2.2 アプリケーション層

**表 3.2.2-1 アプリケーション層の仕様**

| 項番 | 項目               | 適用 |
|------|--------------------|------|
| 1    | プロトコル         | AMQP 1.0（ISO/IEC 19464:2014 準拠） |
| 2    | 使用インタフェース | Java Message Service（JMS）API |
| 3    | ソケット接続方式   | メッセージを受信する場合は受信専用のコネクション、メッセージを送信する場合は送信専用のコネクションを接続する。各コネクションの接続は、SWIM サービス利用者システムから本システムに対して行われる。 |
| 4    | メッセージ転送方式 | 「3.2.3 メッセージフォーマット」「3.2.4 通信シーケンス」により規定する。 |
| 5    | 文字コード         | 制御ヘッダ：UTF-8／業務メッセージ本文：各付録を参照。 |
| 6    | 無通信タイムアウト | 1 時間 ※AMQP セッション確立後、無通信の状態が 1 時間以上続いた場合、AMQP セッションが強制的に切断される。この場合、再度セッションを確立する必要がある。 |

AMQP では、メッセージ転送モデル（1 対 1 通信モデル）と Pub/Sub モデル（1 対多通信モデル）の 2 つの通信モデルをサポートする。

**表 3.2.2-2 メッセージ転送モデルの概要説明**

| 項番 | 名称                                   | 概要説明 |
|------|----------------------------------------|----------|
| 1    | 受信 Queue（AMQP ブローカー）          | SWIM サービス提供システムから転送されたメッセージを一時的に保管する。基本的には FIFO により処理される。 |
| 2    | 送信 Queue（AMQP ブローカー）          | SWIM サービス提供システムへ転送するメッセージを一時的に保持する。基本的には FIFO により処理される。 |
| 3    | 利用者専用プログラム（サービスアセンブリ） | SWIM サービス利用者専用プログラム。送信 Queue に転送されたメッセージを解析し、適切な宛先の受信 Queue へ転送する。 |

**表 3.2.2-3 Pub/Sub モデルの概要説明**

| 項番 | 名称                                   | 概要説明 |
|------|----------------------------------------|----------|
| 1    | 受信 Queue（AMQP ブローカー）          | SWIM サービス利用者システムへ配信するメッセージを一時的に保管する。基本的には FIFO により処理される。 |
| 2    | Topic（AMQP ブローカー）               | SWIM サービス利用者へ一斉配信するメッセージを一時的に保持する。基本的には FIFO により処理される。 |
| 3    | 利用者専用プログラム（サービスアセンブリ） | Topic に配信されたメッセージを解析し、利用者の受信 Queue へ転送すべきか判断し、必要なら受信 Queue へ転送する。 |

#### 3.2.3 メッセージフォーマット

「Java Message Service（JMS）API」のメッセージは、ヘッダとプロパティとボディから構成される。AMQP におけるメッセージフォーマットのフィールドは、JMS における論理レイアウトの各フィールドがマッピングされる。

- AMQP クライアントのプログラムでは**ヘッダ値を変更せず**、デフォルト値を使用する。（※JMSMessageID, JMSTimestamp, JMSCorrelationID などの定義あり）
- AMQP クライアントのプログラムでは**プロパティ値を変更せず**、デフォルト値を使用する。（※JMSXUserID, JMSXAppID などの定義あり）

アプリケーション間で交換するデータを格納する。ボディのフォーマットについては、各付録に規定する。

#### 3.2.4 通信シーケンス

SWIM サービス利用者は、メッセージ送受信用のセッションを持つ。セッション確立は、SWIM サービス利用者から本システムに対して行い、完了するとメッセージの送受信が可能となる。

セッション切断は、SWIM サービス利用者、本システムのどのシステムからも行うことができる。

- **【本システムから SWIM サービス利用者へ】** 送信 Queue に格納されたメッセージが解析され、利用者の受信 Queue に転送され、JMS を利用して取り出される。
- **【SWIM サービス利用者から本システムへ】** 利用者が JMS を利用して送信 Queue にメッセージを転送し、プログラムにより解析され処理される。

本システムが Topic に一斉配信するメッセージを保持し、条件に合致したものが各利用者の受信 Queue に転送される。

- **本システム側の障害:** エラー発生を契機として検知。セッション再確立を試みる必要がある。
- **クライアント側の障害:** クライアント側で復旧後、セッション再確立を試みる必要がある。

## 別紙 2 AMQP クライアントサンプル

### メッセージ受信処理（AMQP）

```java
package org.apache.activemq.artemis.jms.example;

import java.util.Properties;
import javax.jms.Connection;
import javax.jms.ConnectionFactory;
import javax.jms.MessageConsumer;
import javax.jms.Session;
import javax.jms.Destination;
import javax.jms.TextMessage;
import javax.naming.Context;
import javax.naming.InitialContext;

public class AMQPQueueExampleRecv {
    public static void main(String[] args) throws Exception {
        Properties props = new Properties();
        props.setProperty(Context.INITIAL_CONTEXT_FACTORY,
        "org.apache.qpid.jms.jndi.JmsInitialContextFactory");

        // ブローカーURL を指定
        props.setProperty("connectionFactory.F_R000000000001",
        "amqp://10.20.30.40:5672");

        // 接続先の受信キューID を指定（「::」の前後に同じ受信キューID を設定する。）
        props.setProperty("queue.R.000000000001", "R.000000000001::R.000000000001");

        props.setProperty("FACKEY", "F_R000000000001");
        props.setProperty("RECVKEY", "R.000000000001");

        // Step 1. Create an initial context to perform the JNDI lookup.
        javax.naming.Context context = new InitialContext(props);
        String fackey = (String)props.getProperty("FACKEY");
        String recvkey = (String)props.getProperty("RECVKEY");

        // Step 2. Look-up the JMS topic
        ConnectionFactory connectionFactory = (ConnectionFactory) context.lookup(fackey);
        Destination r_queue = (Destination) context.lookup(recvkey);

        Connection connection = null;
        Session session = null ;
        MessageConsumer consumer = null;
        boolean exitflg = false;

        while(false == exitflg) {
            try {
                if(null == connection) {
                    // Step 3. Look-up the JMS connection factory
                    // キュー接続用認証 ID、キュー接続用パスワードを設定する。
                    connection = connectionFactory.createConnection("id","password");

                    // Step 4. Create a transactional JMS session
                    session = connection.createSession(true, Session.AUTO_ACKNOWLEDGE);

                    // Step 5. Start the connection
                    connection.start();

                    // Step 6. Create a message consumer
                    consumer = session.createConsumer(r_queue);
                }

                // Step 7. Receive the message (メッセージ構成は各サービスの各付録を参照)
                TextMessage tm = (TextMessage) consumer.receive();

                // Step 8. Commit the session
                session.commit();

                // 受信処理を終了させる場合は exitflg に true を代入する。
                //exitflg = true;

            } catch(Exception ex) {
                if(null != connection) {
                    // Step 9. Close the connection
                    connection.close();
                    connection = null;
                }
                Thread.sleep(1000);
            }
        }
    }
}
```
